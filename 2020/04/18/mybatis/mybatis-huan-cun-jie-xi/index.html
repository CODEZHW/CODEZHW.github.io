前言

在简单介绍了Mybatis的创建以及执行过程之后，我们想要了解的是Mybatis里的缓存机制，Mybatis有两种缓存级别分别为一级和二级：

一级是默认开启的：一级缓存是SqlSession级别的缓存。执行相同的sql语句的情况下，我们使用同一个对象调用的同一个XXXMapper的方法，只会执行一次SQL，在这一次SQL执行完之后会将结果缓存，再以后的查询中如果没有声明刷新缓存，并且缓存没有超时的情况下，Mybatis不会再去请求数据库，而是直接调用缓存的结果。

二级缓存需要我们手动配置：二级缓存是Application级别的缓存。它可以实现多个SqlSession之间的数据共享，怎样去实现多个SqlSession之间的数据共享呢？，我们都知道每一个SqlSession是由SqlSessionFactory工厂创建出来的，那么原理就是Mybatis中SqlSessionFactory对象的缓存，同一个SqlSessionFactory创建的SqlSession对象共享缓存。

一级缓存：从流程看源码

从流程看源码，每一个SqlSession都持有一个Executor。

Executor： SqlSession向用户提供操作数据库的方法，但和数据库操作有关的职责都会委托给Executor。

如下图所示，Executor有若干个实现类，为Executor赋予了不同的能力，大家可以根据类名，自行学习每个类的基本作用。

每一个Executor中持有一个LocalCache对象,它是一个PerpetualCache类型的缓存对象，它具体实现是一个HashMap。

所以一级缓存主要是通过对HashMap的操作来实现的，如果HashMap中没有数据，那么就执行查询数据库的操作。

既然缓存是利用HashMap的结构来实现的，那么它的value是查询到的数据，key是什么呢？(select()的本质还是queru()操作)

我们从源码中得知，key是一个CacheKey类型的对象，封装一个CacheKey主要利用到了MappedStatement对象的ID属性（还有其他参数），从创建流程的源码中得知，Mybatis将执行的sql语句封装成MappedStatement对象

这也印证了以 一条sql对应一个结果这样的形式存储在HashMap中。

我们这里提到了PerpetualCache类型的缓存，那我们就需要了解以下Mybatis中的缓存类型有哪些

Cache：MyBatis中的Cache接口，提供了和缓存相关的最基本的操作

Cache有若干个实现类，使用装饰器模式互相组装，提供丰富的操控缓存的能力，部分实现类如下图所示：

可以从名字去简单了解到它们的作用，其中weak和soft类型的缓存可以理解为JVM中的软弱引用，它们的生命周期是类似的。

private SqlSession openSessionFromDataSource(ExecutorType execType, TransactionIsolationLevel level, boolean autoCommit) {
    Transaction tx = null;
    try {
      final Environment environment = configuration.getEnvironment();
      final TransactionFactory transactionFactory = getTransactionFactoryFromEnvironment(environment);
      tx = transactionFactory.newTransaction(environment.getDataSource(), level, autoCommit);
        //创建我们的默认执行器
      final Executor executor = configuration.newExecutor(tx, execType);
        //重点
      return new DefaultSqlSession(configuration, executor, autoCommit);
    } catch (Exception e) {
      closeTransaction(tx); // may have fetched a connection so lets call close()
      throw ExceptionFactory.wrapException("Error opening session.  Cause: " + e, e);
    } finally {
      ErrorContext.instance().reset();
    }
  }

那么问题来了：既然cacheEnabled是默认开启的，executor应该是CachingExecutor类型的缓存对象啊。

我们进入CachingExecutor查看它的query()方法

public <E> List<E> query(MappedStatement ms, Object parameterObject, RowBounds rowBounds, ResultHandler resultHandler, CacheKey key, BoundSql boundSql)
    throws SQLException {
    //它会首先判断MappedStatement对象是否存在cache，如果没有就利用装饰器去执行query。
  Cache cache = ms.getCache();
  if (cache != null) {
    flushCacheIfRequired(ms);
    if (ms.isUseCache() && resultHandler == null) {
      ensureNoOutParams(ms, boundSql);
      @SuppressWarnings("unchecked")
      List<E> list = (List<E>) tcm.getObject(cache, key);
      if (list == null) {
        list = delegate.query(ms, parameterObject, rowBounds, resultHandler, key, boundSql);
        tcm.putObject(cache, key, list); // issue #578 and #116
      }
      return list;
    }
  }
  return delegate.query(ms, parameterObject, rowBounds, resultHandler, key, boundSql);
}

很显然，我们是一级缓存没有配置cache所以，它归根结底还是执行的SimpleExecutor.query().

至此我们就了解到了：DefaultSqlSession->持有->SimpleExecutor->调用父类->BaseExecutor.query()方法。

在查询过程中，它会去localCache判断sql语句对应的结果是否存在，如果存在直接处理数据；如果不存在，去查询数据库。

在查询的最后它还会判断LocalCacheScope如果是STATEMENT的话，它会清空缓存；默认的LocalCacheScope是SESSION。

再来看一下为什么执行更新操作会使缓存失效。

@Override
public int update(MappedStatement ms, Object parameter) throws SQLException {
  ErrorContext.instance().resource(ms.getResource()).activity("executing an update").object(ms.getId());
  if (closed) {
    throw new ExecutorException("Executor was closed.");
  }
  clearLocalCache();
  return doUpdate(ms, parameter);
}

在执行doUpdate()方法之前，它会执行一次clearLocalCache()的操作，清空缓存。

总结：

  1. MyBatis一级缓存的生命周期和SqlSession一致。
  2. MyBatis一级缓存内部设计简单，只是一个没有容量限定的HashMap，在缓存的功能性上有所欠缺。
  3. MyBatis的一级缓存最大范围是SqlSession内部，有多个SqlSession或者分布式的环境下，数据库写操作会引起脏数据，建议设定缓存级别为Statement。

二级缓存的过程

二级缓存是默认不开启的，要启动Mybatis的二级缓存需要以下几个条件：

  • 实体类必须是可序列化的
  • XXXMapper开启缓存

在XXXMapper中开启缓存默认配置如下：

  • 映射语句文件中的所有 select 语句的结果将会被缓存。
  • 映射语句文件中的所有 insert、update 和 delete 语句会刷新缓存。
  • 缓存会使用最近最少使用算法（LRU, Least Recently Used）算法来清除不需要的缓存。
  • 缓存不会定时进行刷新（也就是说，没有刷新间隔）。
  • 缓存会保存列表或对象（无论查询方法返回哪种）的 1024 个引用。
  • 缓存会被视为读/写缓存，这意味着获取到的对象并不是共享的，可以安全地被调用者修改，而不干扰其他调用者或线程所做的潜在修改。

我们可以手动的更改默认的配置。

编写一个测试类

@Test
public void testCache () {
    UserDao userDao1 = sqlSession.getMapper ( UserDao.class );
    UserDao userDao2 = sqlSession1.getMapper ( UserDao.class );

    List <User> users = userDao1.QueryAll ( );
    for (User user : users) {
        System.out.println ( user );
    }

    sqlSession.close ();

    List <User> users1 = userDao2.QueryAll ( );
    for (User user : users1) {
        System.out.println ( user );
    }
    sqlSession1.close ();
}

可以看到我们在关闭了sqlSession之后利用sqlSession1进行查询，也能拿到缓存的结果，这就是二级缓存多个Session之间可以共享缓存数据，缓存命中率是0.5。
