前言

加载是类加载机制中的一个重要的过程，它的主要目的是通过一个类的全限定类名把该类的class文件加载到虚拟机中,这个操作被放在虚拟机外部，可以让用户自己去决定想要加载的类，这为Java的快速流行和广泛应用做出了贡献。 实现这个动作的实际’操作者’就是类加载器(ClassLoader)。

类与类加载器

类加载器虽然只用于实现类的加载动作，但它在Java程序中起到的作用却远超类加载阶段。

从概念上理解：既然类加载器是用来加载类的，那么一个类在虚拟机中一定对应着一个类加载器。这个类以及它的类加载器确定了这个类在虚拟机中的唯一性。

需要注意的是：比较两个对象是否’相等’，是建立在它们的类加载器一样的前提下的。

如果两个类的类加载器不一致，那怕它们是属于同一个Class文件，它们也不会相等。

如果没有注意到类加载器的影响，在某些情况下可能会产生具有迷惑性的结果。

我们自定义一个类加载器并创建类的实例，去与应用程序类加载器加载的类进行比较，看它们是否相等。

ppublic class ClassLoaderTest {
    public static void main ( String[] args ) throws ClassNotFoundException, IllegalAccessException, InstantiationException, NoSuchMethodException, InvocationTargetException {
        Object object = null;
        //获取通过类加载器创建的Class对象
        Class clazz = MyClassLoader.getMyClassLoader ( ).findClass ( "dynamic.test.ClassLoaderTest" );
        //利用反射创建对象
        if (clazz != null) {
            object = clazz.getDeclaredConstructor ( ).newInstance ( );
        }

        System.out.println ( "object的类加载器: " + object.getClass ( ).getClassLoader ( ) );
        System.out.println ( "ClassLoaderTest的类加载器: " + ClassLoaderTest.class.getClassLoader ( ) );
        System.out.println ( "----------------------------------" );
        System.out.println ( "object的class: " + object.getClass ( ) );
        System.out.println ( "ClassLoaderTest的class: " + ClassLoaderTest.class);

        System.out.println ( object instanceof dynamic.test.ClassLoaderTest );


    }

}

class MyClassLoader extends ClassLoader {

    static MyClassLoader getMyClassLoader () {
        return new MyClassLoader ( );
    }

    @Override
    public Class <?> findClass ( String name ) throws ClassNotFoundException {
        String fileName = name.substring ( name.lastIndexOf ( "." ) + 1 ) + ".class";
        InputStream inputStream = getClass ( ).getResourceAsStream ( fileName );
        byte[] bytes = new byte[0];
        try {
            //jdk9才有的readAllBytes()方法
            bytes = inputStream.readAllBytes ( );
        } catch (IOException e) {
            e.printStackTrace ( );
        }
        //返回通过这个类加载器创建的Class对象
        return defineClass ( name, bytes, 0, bytes.length );
    }
}

我们通过重写ClassLoader::findClass()方法自定义一个类加载器MyClassLoader，通过这个类加载器和反射机制去创建一个ClassLoaderTest对象object，然去比对它和应用程序类加载器创建的对象是否相等。

object对象和ClassLoaderTest类的Class对象是一致的；但是它们的类加载器不同，object是我们自己定义的类加载器，ClassLoaderTest类的类加载器是应用程序类加载器；它们的类加载器不一致，所以它们两的对象不’相等’。

双亲委派模型

这里直接从网上找的图

从图中可以大致看出，我们的双亲委派模型主要由三个部分构成：

  • 启动类加载器（Bootstrap Class Loader): 主要负载加载存放在\lib目录下的文件，或者被

    -Xbootclasspath参数所指定的路径中存放的，而且是Java虚拟机能够识别的（rt.jar、tools.jar，名字不符合的类库即使放在lib目录中也不会被加载）类库加载到虚拟机的内存中。这个加载器用户是无法引用的，它是由C++编写的。

  • 扩展类加载器（Extension Class Loader): 主要负责加载存放在\lib\ext目录中，或者被java.ext.dirs系统变量所指定的路径中所有的类库。

  • 应用程序类加载器（Application Class Loader）：这个类加载器sun.misc.Launcher$AppClassLoader来实现。由于应用程序类加载器是ClassLoader类中的getSystem-ClassLoader()方法的返回值，所以有些场合中也称它为“系统类加载器”。它负责加载用户类路径（ClassPath）上所有的类库，开发者同样可以直接在代码中使用这个类加载器。

我们可以通过System.getProperty ( "java.class.path" )去查看当前路径下加载的jar包

可以看到lib/ext下的文件被加载; lib/的文件被加载；我们自己定义的类也被加载进来了。

除了上诉这三种类加载器，用户还可以自定义类加载器，用来加载从磁盘之外的Class文件来源，Tomcat就利用了这点。

自此我们就可以真正的介绍双亲委派模型的概念了：

如果一个类的类加载器收到了类加载的请求，首先它不会直接加载；而是把这个类的加载请求委托给父类去加载，如果父类无法完成加载还会继续这个过程直至委托给最顶层的类加载器去加载，如果所有父类都无法完成加载，那么子类才会尝试自己去加载。

使用双亲委派模型的好处：

前面已经提到过，在由多个类加载器加载同一个类的时候，会造成视觉混乱。所以它的好处在于将类委托给父类去加载确保了不同类之间的层次关系，比如说有些是Java核心功能的类(Object)，我们希望由最顶层的类加载器去加载，这样这个Object在整个程序运行期间都能确保是同一个类，保证了稳定性。

protected Class<?> loadClass(String name, boolean resolve)
    throws ClassNotFoundException
{
    synchronized (getClassLoadingLock(name)) {
       //检查类是否被加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            long t0 = System.nanoTime();
            try {
                if (parent != null) {
                    //如果找不到，则委托给父类加载器去加载
                    c = parent.loadClass(name, false);
                } else {
                    //如果没有父类，则委托给启动加载器去加载
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
            }

            if (c == null) {
                long t1 = System.nanoTime();
                // 如果都没有找到，则通过自定义实现的findClass去查找并加载
                c = findClass(name);

                // this is the defining class loader; record the stats
                PerfCounter.getParentDelegationTime().addTime(t1 - t0);
                PerfCounter.getFindClassTime().addElapsedTimeFrom(t1);
                PerfCounter.getFindClasses().increment();
            }
        }
        if (resolve) {
            //是否需要在加载时进行解析
            resolveClass(c);
        }
        return c;
    }
}

很好的印证了我们的双亲委派流程。

在JDK1.2之前，在自定义类加载时，总会去继承ClassLoader类并重写loadClass方法，从而实现自定义的类加载类，但是在JDK1.2之后已不再建议用户去覆盖loadClass()方法，而是建议把自定义的类加载逻辑写在findClass()方法中。从前面的分析得知findClass()是在所有父类加载器都无法加载对象的前提下，调用来确保自定义的类加载器符合双亲委派模型。

破坏双亲委派模型

双亲委派模型并不是一个具有强制性约束的模型，而是Java设计者推荐给开发者们的类加载器实现方式。

  • 由于双亲委派模型是JDK1.2之后引入的，但是类加载器的概念和抽象类java.lang.ClassLoader则在Java的第一个版本中就已经存在，面对已经存在的用户自定义类加载器的代码，Java设计者们引入双亲委派模型时不得不做出一些妥协，为了兼容这些已有代码，无法再以技术手段避免loadClass()被子类覆盖的可能性，只能在JDK 1.2之后的java.lang.ClassLoader中添加一个新的protected方法findClass()，并引导用户编写的类加载逻辑时尽可能去重写这个方法，而不是在loadClass()中编写代码。
  • 线程上下文类加载器。
  • 双亲委派模型的第三次“被破坏”是由于用户对程序动态性的追求而导致的，这里所说的“动态性”指的是一些非常“热”门的名词：代码热替换（Hot Swap）、模块热部署（HotDeployment）等。

简单实现一个热部署程序

主要思想：通过自定义类加载器与多线程监控文件是否修改(重写编译)来实现热部署。

步骤：

  • 创建一个任务类模拟业务中的修改实现热部署

  • 自定义类加载器

  • 创建一个守护线程去监控任务类的.class是否进行了修改即重新编译，如果发生了我们就通过自定义的类加载器去重新加载这个类。

  • 任务类
public class Task implements Runnable {

    @Override
    public void run () {
        System.out.println ( Thread.currentThread ( ).getName ( ) + "：这是为修改之前的程序！！！" );
    }

}

  • 自定义的类加载器

    class MyClassLoader extends ClassLoader {

    /**
     * 静态工厂方法
     * @return 自定义的类加载器
     */
    public static MyClassLoader getMyclassLoader () {
        return new MyClassLoader ( );
    }

    /**
     * 重写了findClass()方法实现自定义类加载器
     * @param name 类的路径
     * @return 类的Class对象
     * @throws ClassNotFoundException
     */
    @Override
    public Class <?> findClass ( String name ) throws ClassNotFoundException {
        try {
            String fileName = "/" + name.replaceAll ( "\\.", "/" ) + ".class";
            InputStream is = getClass ( ).getResourceAsStream ( fileName );
            byte[] bytes = is.readAllBytes ( );
            return defineClass ( name, bytes, 0, bytes.length );
        } catch (IOException e) {
            throw new ClassNotFoundException ( name );
        }
    }

}

  • 主要程序

class HotDeploy {
    private static volatile Runnable instance;
    private final String FILE_NAME;  //Class文件的全限定类名
    private final String CLASS_NAME; //任务类的名称

    /**
     * 构造函数
     * @param name 文件名
     */
    public HotDeploy ( String name ) {
        CLASS_NAME = name;
        name = name.replaceAll ( "\\.", "/" ) + ".class";
        FILE_NAME = (getClass ( ).getResource ( "/" ) + name).substring ( 6 );
    }


    /**
     *  DCL单例模式创建一个Runnable实例,单例模式确保一个类加载器只创建除一个对象
     * @return
     */
    public Runnable getTask () {
        if (instance == null) {
            synchronized (HotDeploy.class) {
                if (instance == null) {
                    try {
                        instance = createTask ( );
                    } catch (Exception e) {
                        e.printStackTrace ( );
                    }
                }
            }
        }
        return instance;
    }

    /**
     * 监视器，开启一个线程去监视class文件是否被修改过；如果修改了，就重新加载这个类。
     * @throws IOException
     */
    public void monitor () throws IOException {

        Thread t = new Thread ( () -> {
            try {
                //获取class文件的最后修改时间
                long lastModified = Files.getLastModifiedTime ( Path.of ( FILE_NAME ) ).toMillis ( );
                while (true) {
                    TimeUnit.SECONDS.sleep ( 3 );
                    long now = Files.getLastModifiedTime ( Path.of ( FILE_NAME ) ).toMillis ( );
                    //判断class文件是否修改过
                    if (now != lastModified) {
                        lastModified = now;
                        //修改过就重新创建一个实例
                        instance = createTask ( );
                    }
                }
            } catch (InterruptedException | IOException e) {
                e.printStackTrace ( );
            }
        } );
        t.setDaemon ( true );
        t.start ( );
    }


      /**
     * 通过类加载器得到的Class对象，利用反射机制创建对象。
     * @return
     */
    private Runnable createTask () {
        try {
            //获取生成的Class对象
            Class clazz = MyClassLoader.getMyclassLoader ( ).findClass ( CLASS_NAME );
            if (clazz != null) {
                //利用反射创建对象实例
                return (Runnable) clazz.getDeclaredConstructor ( ).newInstance ( );
            }
        } catch (Exception e) {
            e.printStackTrace ( );
        }
        return null;
    }

}
  • main方法
    public static void main ( String[] args ) throws InterruptedException, IOException {
  HotDeploy hotDeploy = new HotDeploy ( "dynamic.Task" );
  hotDeploy.monitor ( );
  while (true) {
      TimeUnit.SECONDS.sleep ( 3 );
      hotDeploy.getTask ( ).run ( );
  }
}

可以看到我们修改task程序，然后重新编译实现了热部署。
