前言

这是一道面试题： 写一段代码在遍历 ArrayList 时移除一个元素？

考察的目的是：使用正确的方式来实现在遍历的过程中移除元素，而不会出现 ConcurrentModificationException 异常的示例代码。

今天从源码的角度去解析这个问题，分析一下ArrayList数组。

案例

这里有一段代码：实现了遍历list时，找到值为s4的元素，然后删除掉它。

ArrayList <String> list = new ArrayList <> ( );
        list.add ( "s1" );
        list.add ( "s2" );
        list.add ( "s3" );
        list.add ( "s4" );

        Iterator <String> iterator = list.iterator ( );

        while (iterator.hasNext ( )) {
            if (iterator.next ( ).equals ( "s4" )) {
                list.remove ( "s4" );
            }
        }

可以看到它抛出了一个多线程修改异常。

从报错来看我们可以知道它出错的问题在于，ArrayList->Itr(内部类)-chekForComdification（方法）。

我们先修改一下代码，我们删除第三个元素s3。

while (iterator.hasNext ( )) {
            if (iterator.next ( ).equals ( "s3" )) {
                list.remove ( "s3" );
            }
        }

尽然发现删除成功了!我们逐一尝试s1,s2,s3,s4，发现除了s3以外的元素都会抛出异常。我们增大链表元素到7，发现发现除了s6以外的元素都会抛出异常。这时候可以发现规律是，只有删除倒数第二个元素的时候不会抛出异常，删除其他元素都会抛出异常。

我们先点进ConcurrentModificationException类来看一下它的说明。

  • 直接看文档注释：
This exception may be thrown by methods that have detected concurrent
modification of an object when such modification is not permissible.

一个方法检测到被一个对象多线程修改时，会抛出这个异常。

For example, it is not generally permissible for one thread to modify a 
Collection while another thread is iterating over it.

举例：一个线程对集合进行修改，另一个线程对集合进行遍历。

Some Iterator implementations (including those of all the general purpose collection implementations provided by the JRE) may choose to throw this exception if this behavior is detected.

一但检测到这个，一些Iterator的实现类可以能抛出这个异常。

Iterators that do this are known as <i>fail-fast</i> iterators。

这样的迭代器我们称它为-快速失败迭代器。

Note that this exception does not always indicate that an object has been concurrently modified by a <i>different</i> thread.  If a single thread issues a sequence of method invocations that violates the contract of an object, the object may throw this exception.  For example, if a thread modifies a collection directly while it is iterating over the collection with a fail-fast iterator, the iterator will throw this exception.

请注意，这个异常并只是再多线程下进行修改时候会产生，在单线程环境下，如果你使用了迭代器对集合进行修改有可能产生这个异常。

ArrayList源码分析

  • 关键字段

//AbstractList里面的字段，ArrayList继承自AbstractList，用于记录数组结构变化的次数。
protected transient int modCount = 0;

//数组的默认容量
private static final int DEFAULT_CAPACITY = 10;

//空数组
private static final Object[] EMPTY_ELEMENTDATA = {};

//空数组 用于初始化
private static final Object[] DEFAULTCAPACITY_EMPTY_ELEMENTDATA = {};

//数组缓存区  当添加第一个元素的时候elementData == DEFAULTCAPACITY_EMPTY_ELEMENTDATA将会被转化为EMPTY_ELEMENTDATA
transient Object[] elementData; 

//数组元素个数
private int size;
  • 案例分析

Itr内部类分析

int expectedModCount = modCount;
public boolean hasNext() {
    //cursor下一个元素的索引
    //当前容器的值
    return cursor != size;
}

@SuppressWarnings("unchecked")
public E next() {
    //很关键: 对modCount检查
    checkForComodification();
    int i = cursor;
    if (i >= size)
        throw new NoSuchElementException();
    Object[] elementData = ArrayList.this.elementData;
    if (i >= elementData.length)
        throw new ConcurrentModificationException();
    cursor = i + 1;
    return (E) elementData[lastRet = i];
}

final void checkForComodification() {
    //如果当前的操作次数与预期的操作次数不相等，则会抛出异常。
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}

那么很明显了，正式因为list.move（）操作导致了modCount != expectedModCount，所以抛出异常。

  • list.move()源码

    public boolean remove(Object o) {
  final Object[] es = elementData;
  final int size = this.size;
  int i = 0;
  found: {
      if (o == null) {
          for (; i < size; i++)
              if (es[i] == null)
                  break found;
      } else {
          for (; i < size; i++)
              if (o.equals(es[i]))
                  break found;
      }
      return false;
  }
  //上面的操作都是从数组中，找到我们想要删除的元素。
  //真正执行删除操作的是 fastRemove
  fastRemove(es, i);
  return true;
}

private void fastRemove(Object[] es, int i) {
//就是这个操作。
modCount++;
final int newSize;
if ((newSize = size - 1) > i)
System.arraycopy(es, i + 1, es, i, newSize - i);
es[size = newSize] = null;
}




**现在来开最开始的问题：为什么到数第二个元素可以被删除？**

抛出异常的位置是在：`next()->checkForComodification()-modCount != expectedModCount`判断条件不相等。

那难道删除倒数第二个元素，它们的值就会相等吗？

不会的，你只要执行了list.remove()操作都会使modCount++，而且expectedModCount是在创建Ttor对象的时候赋值为modCount，所以它们是不相等的。

那是怎么回事呢？

我们循环的条件是：`while(iterator.hasNext ())=> return cursor != size;` 这个判断条件

我们可以看到在fastRemove操作里面有一个`es[size = newSize] = null;`操作，这个操作使得size = size-1

而`cursor`是指向下一个元素的索引，假设当前list中有三个元素，遍历到了第二个元素此时cursor=2（从0起，指向的是下一个元素也就是数组中的第三个元素），此时执行remove操作后，`size = 3 - 1 = 2`在下一次

hasNext()判断的时候就直接返回了，不会执行next()方法了,所以不会抛出异常。

**怎么样解决这个问题呢？**

**我们可以调用iterator.move()方法。**

```java
public void remove() {
    if (lastRet < 0)
        throw new IllegalStateException();
    checkForComodification();

    try {
        //调用的是重写的remove方法 重写自AbstractList，继承List
        ArrayList.this.remove(lastRet);
        cursor = lastRet;
        lastRet = -1;

        expectedModCount = modCount;
    } catch (IndexOutOfBoundsException ex) {
        throw new ConcurrentModificationException();
    }
}

public E remove(int index) {
        Objects.checkIndex(index, size);
        final Object[] es = elementData;

        @SuppressWarnings("unchecked") E oldValue = (E) es[index];
        //虽说本质也是调用了fastRemove，但是它的预期操作数也同步跟新了，所以不会抛出异常;
        fastRemove(es, index);

        return oldValue;
    }

最来再来简单聊一聊ArrayList的其他源码

任何源码都从构造函数起

//其实什么也没有，就是初始化一个空的数组~ 
//这个赋值的条件，在后面真正初始化的时候会用到~
public ArrayList() {
    this.elementData = DEFAULTCAPACITY_EMPTY_ELEMENTDATA;
}

ArrayList数组的真正初始化发生在add()方法执行的阶段。

为什么要这样做了？我认为是为了方便数组的扩容，所以把它们写在了一起。

public boolean add(E e) {
    modCount++;
    add(e, elementData, size);
    return true;
}

private void add(E e, Object[] elementData, int s) {
        //判断一下当前数组的size与数组的size是否相等，初始化肯定相等啊，它们默认都是0；
        //如果数组元素满了也是一样就执行扩容
        if (s == elementData.length)
            elementData = grow();
        elementData[s] = e;
        size = s + 1;
    }


private Object[] grow() {
        return grow(size + 1);
    }


private Object[] grow(int minCapacity) {
        return elementData = Arrays.copyOf(elementData,
                                           newCapacity(minCapacity));
    }

private int newCapacity(int minCapacity) {
        // overflow-conscious code
        //初始为0
        int oldCapacity = elementData.length;
        //扩容为原来的1.5倍
        int newCapacity = oldCapacity + (oldCapacity >> 1);
        // 0 -1 < 0
        if (newCapacity - minCapacity <= 0) {
            //满足构造函数的条件，为数组初始化容量为10.
            if (elementData == DEFAULTCAPACITY_EMPTY_ELEMENTDATA)
                return Math.max(DEFAULT_CAPACITY, minCapacity);
            if (minCapacity < 0) // overflow
                throw new OutOfMemoryError();
            return minCapacity;
        }
        return (newCapacity - MAX_ARRAY_SIZE <= 0)
            ? newCapacity
            : hugeCapacity(minCapacity);
    }


private static int hugeCapacity(int minCapacity) {
    if (minCapacity < 0) // overflow
        throw new OutOfMemoryError();
    return (minCapacity > MAX_ARRAY_SIZE)
        ? Integer.MAX_VALUE
        : MAX_ARRAY_SIZE;
}

//在指定位置添加元素的方法
public void add(int index, E element) {
        rangeCheckForAdd(index);
        modCount++;
        final int s;
        Object[] elementData;
        //判断是否需要扩容
        if ((s = size) == (elementData = this.elementData).length)
            elementData = grow();
        //native本地方法，把从index起往后的元素全部后移，空出idex位置。
        System.arraycopy(elementData, index,
                         elementData, index + 1,
                         s - index);
        elementData[index] = element;
        size = s + 1;
    }

//遍历数组找到要删除的元素i，如果i是最后一个元素直接删除，如果不是将
private void fastRemove(Object[] es, int i) {
        modCount++;
        final int newSize;
        if ((newSize = size - 1) > i)
            System.arraycopy(es, i + 1, es, i, newSize - i);
        es[size = newSize] = null;
    }

删除的实现原理是：将第i的位置之后的所有元素全部往前移，将第i个元素移至数组末尾。

然后删除末尾的元素。

这里有一个问题？为什么要先移动再删除呢？

这是为了提高代码的复用，这样可以同时将要删除的数组元素再最后一个这样的情况考虑进来。

会减少一些不必要的代码。

总结

JDK1.8之前数组的默认初始化容量是10，JDK1.8之后则是0。

JDK1.8的默认容量是10，但是并不是在初始化的时候就为数组赋值了，在ADD（）方法第一次执行的时候为数组初始化容量

通过增强for循环的方式遍历容器，并移除同样会报并发修改异常。

因为增强for循环被JVM解析出来后实际上就是Iterator。
